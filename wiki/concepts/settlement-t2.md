# T+2 交割機制

## 基本定義

台灣證券市場採用 **T+2 交割**（Trade Date + 2 Business Days）：

```
T（日）：     買進成交日。股票暫未到帳，帳面顯示成本。
T+1（次日）： 券商處理交割準備。
T+2（交割日）：款項實際從帳戶扣繳，股票正式進入集保。
```

## 實務運作（以 2026-04-14 為例）

| 日期 | 星期 | 事件 |
|------|------|------|
| 2026-04-14（二）| T | 錯誤買進 00919、006208 |
| 2026-04-15（三）| T+1 | 帳面仍是錯誤金額，但未扣款 |
| 2026-04-16（四）| T+2 | **真實扣款日**：~217,454 TWD 被扣 |

## Error 88：集保庫存不足

在 T+2 完成前，股票仍在券商墊付狀態，尚未進入集保。

嘗試在此時賣出：
```
Error 88: 集保庫存不足（Insufficient stock in depository）
```

**這不是系統故障，是正常現象。**

```
T → T+2 期間：股票在券商墊付，尚未進集保，無法賣出
T+2 以後：     股票進集保，可以賣出
```

## 帳面餘額 vs 真實餘額

| 資料來源 | T 日顯示 | 實際可用 |
|----------|----------|----------|
| 券商網站/App | 217,900（含墊付）| - |
| 帳戶余額（真實）| ~8,900 | ~8,900 |
| T+2 扣款後 | -209,000 | -209,000 |

**核心原則**：T+2 前，帳面數字不是真正可用的錢。

## 當沖資格的影響

| 資格 | T 日買進 | T+1 賣出 | T+2 取得款項 |
|------|----------|----------|--------------|
| **無當沖** | ✅ 可 | ⚠️ 可（但需集保到帳）| T+2 |
| **有當沖** | ✅ 可 | ✅ 當日可 | 視券商墊付 |

TOMO **未開通當沖**，因此：
- 4/14 買進的股票，4/15 才能尝试卖出
- 但因集保尚未完成，4/15 仍可能失敗（Error 88）
- 最早可能賣出日：**4/16**（T+2 交割完成）

## 對 ETF_TW 系統的影響

### 1. 持倉顯示落後
- `list_positions()` 在 T+2 前不會更新到最新買入
- positions.json 可能顯示錯誤的持股數量
- **解決方案**：以券商 API 即時回傳為準，不要只看 state 檔案

### 2. 現金流計算錯誤
- 帳面餘額包含券商墊付，不是真實可用
- **解決方案**：使用 `api.account_balance().acc_balance` 查真實餘額
  - ⚠️ 注意：Shioaji 顯示的 500,000 是信用額度，非真實現金

### 3. 錯誤監控視窗
- 下單後不能立即假設成交就沒問題
- **解決方案**：
  - 買進後：確認 T+2 帳戶不會被扣到負
  - 賣出後：確認款項 T+2 到帳

## Shioaji 結算 API

```python
api.settlements(api.stock_account)
```

回傳 `SettlementV1` 物件列表，欄位：

| 屬性 | 型別 | 說明 |
|------|------|------|
| `date` | datetime.date | 交割日期 |
| `amount` | float | 交割金額 |
| `T` | int | Tday：T=0（交易日）、T=1（次日）、T=2（交割日）|

可轉換為 DataFrame 便於檢視。T=0/1/2 對應帳面上的預計扣款時程。
