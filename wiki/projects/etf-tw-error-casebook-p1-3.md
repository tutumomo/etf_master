# ETF_TW 錯誤案例庫（P1-3）

## 目的
這份案例庫不是為了記黑歷史，而是把「真實會害人出錯的模式」寫成可反覆引用的錯誤樣板。

每個案例都包含：
- 現象
- 真正根因
- 錯誤推論模式
- 正確規則
- 後續防呆

---

## Case 1：Error 88 / 集保庫存不足

### 現象
- 使用者買入後很快嘗試賣出
- API / 券商回報 Error 88 或集保庫存不足
- 表面看起來像系統壞掉、券商壞掉、或明明有股票卻不能賣

### 真正根因
- 常見根因是 T+2 交割尚未完成
- 也就是：已成交 ≠ 已交割入庫 ≠ 已可自由賣出

### 常見錯誤推論
- 「既然已經買到了，現在就一定賣得掉」
- 「券商餘額有數字，所以庫存一定可賣」
- 「Error 88 是 API 參數錯誤」

### 正確規則
- 先分清楚：成交狀態 / 交割狀態 / 可賣狀態
- 出現 Error 88 時，先查是否仍在 T+2 交割窗口內
- 不要把這類問題直接當成程式 bug

### 防呆做法
- 在制度規則中明寫 T+2
- 在賣出前回答時，主動提醒「已成交不等於已入庫可賣」

## Case 2：假回單 / ghost order

### 現象
- 系統說「已掛單」
- 但券商端查不到
- `orders_open.json` 還殘留 pending / submitted
- `broker_order_id` 是 null 或空字串

### 真正根因
- submit 流程有回應，但缺乏足夠 broker 證據
- 後續 state 沒有把這筆未驗證委託清掉
- dashboard 或 agent 又把殘留 state 當成真的 open order

### 常見錯誤推論
- 「有 submit 回應 = 已正式掛單」
- 「orders_open.json 裡有一筆 = 券商一定有這筆」
- 「pending 就代表委託還在市場上」

### 正確規則
符合以下條件時，視為 ghost order：
- `source_type == submit_verification`
- `verified == false`
- `order_id` 空
- `broker_order_id` 空
- `status in {pending, submitted}`

### 防呆做法
- merge 層直接過濾 ghost order
- dashboard 不把 ghost 單算進未完成委託
- 對外禁止說「✅ 已正式掛單」

## Case 3：state 與 live API 衝突

### 現象
- `positions.json` 顯示有持倉
- live API 顯示數量異常、0 股、1 股，或和券商畫面衝突
- 系統容易為了回答完整，看誰比較像真的就選一個版本

### 真正根因
- state 是 snapshot / 對帳層，不一定等於當下 live truth
- live API 本身也可能有顯示限制或不完整
- 兩邊都不是「不用驗證就能直接宣判」

### 常見錯誤推論
- 「state 有，就是真的」
- 「live API 優先，所以只要查到就一定 100% 正確」
- 「如果兩邊衝突，我挑一邊講就好」

### 正確規則
回答必須分三層：
1. 本次 live API 直接看到
2. 本次 live API 無法確認
3. 次級資訊（state / summary / memory）

### 防呆做法
- 禁止用 state / 記憶補成既成事實
- 關鍵時要求券商 app / web 畫面對帳

## Case 4：Context Summary 當事實

### 現象
- 系統在 handoff / summary 中看到「已下單」「已成交」「已確認」
- 下一輪直接把 summary 當證據引用
- 但實際 live 查詢根本對不上

### 真正根因
- summary 是背景脈絡，不是當次證據
- agent 若偷懶，就會把 summary 當成事實快取

### 常見錯誤推論
- 「summary 寫了，就代表真的發生過」
- 「因為之前說過，所以現在可直接延用」

### 正確規則
- summary 只能提供方向，不可直接當成 live 事實
- 任何『已成功下單 / 已成交 / 現在持有多少』都必須靠當次證據重新確認

### 防呆做法
- 查持倉 / 掛單 / 成交時，先查 live API 或券商畫面
- 若查不到，就直接承認查不到，不用 summary 補洞

## Case 5：股 / 張 / order_lot 混淆

### 現象
- 使用者輸入 100 股
- 系統卻可能送成 1 張
- 或反過來把合法零股錯誤擋掉

### 真正根因
- `Order.quantity`、Shioaji `quantity`、`order_lot` 三者語境不同
- 若只記住「1 張 = 1000 股」，但沒有把語境分清楚，就會犯大錯

### 常見錯誤推論
- 「只要看到 1000，就直接除以 1000」
- 「非 1000 倍數一定非法」
- 「Common / IntradayOdd / Odd 其實差不多」

### 正確規則
- `Order.quantity` 對內一律當股
- `Common`：整股語境，quantity 轉張
- `IntradayOdd`：盤中零股，quantity 用股
- `Odd`：盤後零股，quantity 用股

### 防呆做法
- 一律交給 adapter 實作
- 禁止 agent 在對話中自行心算 lots 後下結論

## Case 6：API 額度 ≠ 真實現金

### 現象
- API 顯示可用數字很大
- 使用者以為帳上真的有那麼多現金
- 實際交易或扣款時才發現不是同一件事

### 真正根因
- 券商 API 顯示的額度、可下單額度、信用額度、真實現金餘額，可能不是同一欄位語義

### 常見錯誤推論
- 「看到餘額欄位，就代表這是現金」
- 「可下單額度 = 現金餘額」

### 正確規則
- 回答時必須區分：
  1. 真實現金
  2. 額度
  3. 可買力

### 防呆做法
- 不把 API 顯示值直接翻譯成『你現在有這麼多現金』

## 錯誤案例庫的使用方式

### 研究 / 查詢時
- 先看是不是落入上述已知錯誤樣板
- 若是，優先套用對應防呆規則

### 正式交易前
- 至少檢查：
  - submit ≠ 落地
  - state ≠ live truth
  - 已成交 ≠ 已入庫可賣
  - quantity 語境是否清楚
  - ghost order 是否已清除

### 對外回答時
- 寧可明說「本次無法確認」
- 也不要用過去事故裡同樣的錯誤推論模式補洞

---

## 最後一句話
每個事故背後，真正危險的都不是 API 壞掉，
而是系統在證據不足時，還假裝自己很確定。
