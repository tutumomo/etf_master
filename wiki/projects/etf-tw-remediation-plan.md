# 台灣 ETF 知識建置與系統健檢計劃書（修正版正式版）

## 一、計劃目標

本計劃不是單純補知識，而是先修正會導致真實交易出錯的根因，再建立可審核、可維護的 ETF 知識與系統基礎。

核心目標分三層：
1. 先消除交易致命錯誤
2. 再校正資料真相源與查詢流程
3. 最後才做知識沉澱、決策優化與長期擴充

## 二、最高原則

1. 不把 state、summary、memory 當成 live 持倉事實
2. 不把 API 單次回傳不足的資料補成「已確認事實」
3. 不把下單邏輯簡化成「股/張換算」這種危險口號
4. 不把家族多角色需求混進單一交易核心
5. 任何真實交易相關敘述，必須可被當次查證

## 三、優先順序重排

- P0：交易防呆與真相源校正
- P1：券商與市場制度知識
- P2：系統審計與決策引擎修正
- P3：使用者知識沉澱與知識庫建置

## 四、正式工作範圍

### P0｜交易防呆與真相源校正（最高優先）

#### P0-1 持倉查詢 SOP 固化
- 每次查持倉時，固定分成四段：
  - 本次 live API 直接看到
  - 本次 live API 無法確認
  - 次級資訊（state / 舊紀錄 / summary，必須明講不是 live）
  - 建議下一步（必要時要求券商 app / web 畫面對帳）
- 禁止用過往記錄倒推出現在持倉
- 禁止在 API 顯示異常時擅自選一個版本當真相

交付物：
- 持倉查詢 SOP 文件
- 技能規則更新
- 對話回覆模板固定化

#### P0-2 掛單 / 成交查詢 SOP 固化
- 僅能使用 Shioaji 可用方法，不得幻想不存在的 API
- 明確標註 `list_trades()` 的可見性限制
- 禁止把「查不到」直接解釋成「一定沒下單」或「一定已成交」
- 掛單查詢與成交查詢要分開敘述，不混為一談

交付物：
- 掛單查詢 SOP
- 成交查詢限制清單
- 回覆模板固定化

#### P0-3 下單安全鏈路校正
- 所有正式送單前，強制使用 ETF_TW/.venv/bin/python
- 一律走 adapter，不可繞過
- pre-flight 一次檢查完畢
- 送單後不能只看 API 回應，必須再做委託驗證
- 不得用口頭推論替代委託落地確認

交付物：
- live submit SOP
- pre-flight 檢查表
- 下單後驗證檢查表

#### P0-4 Shioaji 致命單位風險規則明文化
- Order.quantity 在系統內部以股為主
- 實際整股 / 零股 / order_lot 由 adapter 控制
- 禁止 agent 在對話中自行心算 lots 後直接下結論
- 明確區分整股委託、盤中零股、盤後零股
- 對所有 quantity 顯示欄位加上「語境不保證一致」警告

交付物：
- Shioaji 單位規則文件
- adapter 邏輯檢查清單
- 交易前單位核對規則

#### P0-5 Ghost order 與假回單治理
- 建立 ghost order 判定標準
- 明確定義哪些欄位不足以證明委託已落地
- 清理 state 中殘留假單
- 禁止再出現「已掛單」但券商端不存在的敘述

交付物：
- ghost order 判定規則
- 清理流程
- 驗證流程

### P1｜券商與市場制度知識

#### P1-1 台股 ETF 制度知識
- T+2 結算
- 整股 / 零股交易規則
- 盤中 / 盤後交易時段
- 當沖限制
- ETF 買賣成本與現金流時間差

#### P1-2 Shioaji API 真實行為知識
- `list_trades()` 的實際用途與限制
- `list_positions()` 的顯示風險
- `account_balance()` 與信用額度 / 真實現金差異
- CA 啟用、登入與已知 bug

#### P1-3 交易錯誤案例庫
- Error 88
- 張/股誤解
- 假回單
- state 與 live 不一致
- 以 summary 當事實

### P2｜系統審計與決策引擎修正

#### P2-1 資料流審計
- dashboard、state、live API 的角色分工
- 哪些檔案是展示層
- 哪些檔案是對帳線索
- 哪些欄位不能被視為事實

#### P2-2 訂單監控與填單判定修正
- 修正 fill detection 對 `list_trades()` 的過度依賴
- 建立更保守的監控敘述方式
- 對 pending / filled / failed / unknown 做明確分層

#### P2-3 推薦引擎與決策鏈檢討
- 重新檢查推薦維度是否仍偏誤
- 檢查 reasoning 是否真有內容
- 規則引擎 / AI Bridge / consensus 仲裁是否可審核
- 明確區分「研究候選」與「可執行行動」

#### P2-4 Dashboard 與使用者感知對齊
- 使用者看到的畫面與 agent 說法要一致
- 但一致不代表直接認定為券商真相
- 明確標示 dashboard 是展示層、券商畫面是交易真相層

### P3｜使用者知識沉澱與知識庫建置
- TOMO 投資画像
- 家族需求只保留為背景知識，不進核心交易引擎
- 將制度、錯誤案例、API 限制、SOP 寫入 wiki / skills / memory

## 五、資料來源分級（正式版）

### A 級：當次可驗證真相源
- 本次 live broker API 直接回傳內容
- 使用者當下提供的券商 app / web 畫面

### B 級：對帳與輔助來源
- positions.json
- orders_open.json
- portfolio_snapshot.json
- dashboard 顯示內容

### C 級：背景與脈絡來源
- memory
- 對話 summary
- 舊報告
- 舊截圖
- wiki 既有頁面

規則：
- A 級可直接陳述為「本次查到」
- B 級只能說「次級資訊 / 對帳線索」
- C 級只能說「背景紀錄」，不能說成當下事實

## 六、舊版計劃書需刪除或改寫的句子

- 原句：持倉真相源 — positions.json / orders_open.json 對齊
  - 改為：交易真相源校正 — live API、券商畫面、state 三層分級，不得把 state 當持倉事實

- 原句：永豐金/Shioaji — 下單單位邏輯（股/張）、成交回報機制
  - 改為：永豐金/Shioaji 致命風險規則 — order_lot、整股/零股、查單限制、ghost order、送單後驗證

- 原句：家族需求直接作為核心配置邏輯
  - 改為：家族需求只保留為知識背景，不進核心交易引擎

## 七、執行順序
1. 先完成 P0
2. 再完成 P1
3. 再完成 P2
4. 最後沉澱 P3

## 八、完成標準
1. 再問持倉時，不會再出現用推測補洞的回答
2. 再問掛單時，不會再幻想不存在的 API
3. 再送單時，不會再因單位或 order_lot 出現致命誤判
4. state、dashboard、live API 的角色不再混淆
5. 家族需求不再污染核心交易邏輯
6. 所有重要結論都能追溯到當次證據
