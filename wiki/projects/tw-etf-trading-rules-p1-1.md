# 台股 ETF 交易制度正式規則（P1-1）

## 目的
這份文件只做一件事：把台股 ETF 的制度性規則寫成正式、可引用、可對帳的操作基礎。

它不負責預測漲跌，也不直接證明某筆委託已落地；它負責提供不該搞錯的市場制度。

---

## 1. 交易時段

### 一般盤
- 時間：09:00 - 13:30（台北時間）
- 適用：現股整股交易、盤中零股委託（依券商/下單型態）
- 原則：若不在交易時段內，系統不得假裝可以立即送出一般盤委託

### 盤後零股
- 時間：13:40 - 14:30（台北時間）
- 適用：盤後零股
- 原則：盤後零股與一般盤委託不可混說；回答時要清楚標示是盤中零股還是盤後零股

### 非交易時段
- 其餘時段視為非委託時間
- 原則：非交易時段收到正式下單指令，應直接阻斷，不得用模糊語氣暗示已送單

---

## 2. 整股 / 零股制度

### 整股
- 1 張 = 1000 股
- 若委託數量為 1000 的整數倍，通常屬於整股委託語境
- 在 Shioaji / adapter 實作中，`Common` 類型的 quantity 需注意是以「張」還是券商 API 特定語境送出

### 零股
- 少於 1000 股，或非 1000 整數倍的委託，屬零股語境
- 但系統不得簡化成「非 1000 倍數一定非法」
- 是否可下、怎麼下，必須以當前 adapter 的 `order_lot` 實作為準

### 盤中零股 vs 盤後零股
- 盤中零股：通常對應 `IntradayOdd`
- 盤後零股：通常對應 `Odd`
- 兩者都不是一般整股 `Common`
- 回答與程式都必須區分，不可把兩者統稱成同一種 odd lot 行為

---

## 3. T+2 結算規則

### 買入後何時真正入庫
- T 日買入
- 通常於 T+2 完成交割
- 在交割完成前，券商端可能出現「已買到」與「可自由賣出」之間的時間差

### 為什麼會出現 Error 88 / 集保庫存不足
- 常見原因不是系統壞掉，而是交割尚未完成
- 也就是：你看起來已經買入，但集保庫存尚未真正可供賣出

### 實務原則
- 不能因為「已成交」就直接推論「現在一定可賣」
- 賣出前要把「成交完成」與「交割入庫完成」分開看

---

## 4. 現金、額度、可買力

### 真實現金 vs 顯示額度
- 券商 API 顯示的可用額度，不一定等於真實現金
- 信用額度、可下單額度、帳面數字，可能和實際現金不同

### 正式規則
- 不能把 API 顯示的額度直接說成「你帳上就有這麼多現金」
- 需要時應明確區分：
  1. 真實現金餘額
  2. 券商顯示額度
  3. 可下單上限 / 信用空間

---

## 5. 下單語境正式規則

### quantity 的回答規則
- 對外回答時，必須明講 quantity 是：
  - 股
  - 張
  - 還是某個 API 欄位原樣顯示
- 禁止只說「數量」而不標單位

### order_lot 的回答規則
- 對外若提到 `order_lot`，必須搭配白話翻譯：
  - `Common` = 整股語境
  - `IntradayOdd` = 盤中零股語境
  - `Odd` = 盤後零股語境

### 禁止事項
- 禁止把制度規則講成委託已成功
- 禁止把單位規則講成券商已接受該單
- 禁止把 T+2 規則講成當下 live 持倉證據

---

## 6. 對外回答模板（制度類）

### 當使用者問：現在能不能下單？
回答應包含：
1. 現在是不是交易時段
2. 如果不是，是卡在一般盤還是盤後零股時段
3. 這是制度規則，不等於已送單

### 當使用者問：100股可不可以買？
回答應包含：
1. 100 股屬零股語境
2. 不能用「非 1000 倍數一定拒絕」回答
3. 最終仍要看當前 adapter / 券商 API 的 `order_lot` 實作

### 當使用者問：為什麼不能賣？
回答應先檢查：
1. 是否已成交
2. 是否已交割入庫（T+2）
3. 是否是集保庫存不足 / 額度 / 交易時段問題

---

## 7. 這份文件的邊界

這份文件可以回答：
- 台股 ETF 什麼時段能下單
- 整股 / 零股 / 盤後零股怎麼分
- T+2 是什麼
- 為什麼已成交不等於可賣

這份文件不能單獨回答：
- 你現在實際持有多少
- 某筆單是否已落地
- 某筆單是否真的成交
- 券商 API 此刻是否完整可靠

這些仍必須靠當次 live 查詢或券商畫面證據。
